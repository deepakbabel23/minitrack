The target

Five screens. Nothing invented.

MiniTrack is a minimalist task tracker. You connect with an API key, then view, create, edit, complete, and delete simple tasks.

/connect

Enter an API key (no accounts). Show/hide field, opt-in "remember this session," inline error.

/tasks

Filter All / Active / Completed, task cards, "Load more," inline complete & delete.

/tasks/:id

One task, its badges, and only the actions that exist.

/tasks/new · edit

One shared form for create and edit, with matching validation.

*

A friendly Not-Found with a way back.

The guardrail

A task is only id, title, description, priority, completed. No due dates, assignees, tags, comments — or "Reopen."

Where the build starts

Two inputs feed everything.

🎨 A design system, as CSS

Generated as "MiniTrack Precision" and handed to us finished — we consume it as-is.

  • DESIGN.md — the written spec
  • tokens.css — colors, type, spacing, radius, shadow
  • typography.css — .text-headline-* utilities
  • components.css — .btn · .form-field · .badge · .task-card · .dialog

🔌 A real backend contract

A FastAPI + SQLite JSON API — decoupled, CORS-enabled, key-protected.

  • The app/ layered backend (routes → services → data)
  • The behavioral contract in spec.md
  • Auth via an X-API-Key header
  • Four contract details that will bite if ignored (slide 6)

The method

Every act is one PIRV loop.

We don't make the smallest possible edit. We plan the slice, implement it in the right place, review it against the contract, and verify it runs.

P

Plan

Name the slice and the contract rules it must honor — before any code.

I

Implement

Prompt Claude to build it, with the gotchas baked into the prompt.

R

Review

A read-only frontend-reviewer agent checks it against spec.md.

V

Verify

Run it: dev server, typecheck, lint, and the Vitest suite.

✅
Why it matters hereThe first build discovered the contract's sharp edges the hard way. This time we front-load them, so the live build is right the first time.
Optional aside · design-to-code

It was generated from a one-paragraph brief by a design tool — Google Stitch — and you can prototype the same brief in Figma Make. Same words, two tools; the tokens they produce are exactly what our CSS encodes.

The learnings — front-load these

Four details that bit the first build.

①
422 bodydetail is a plain string, not an array. Never index detail[0].msg.
②
PATCH /tasks/{id}A full replacement — send the whole object; omitted fields reset; it can't change completed.
③
401 bodyHas no request_id (404/422 do). Treat it as optional.
④
ConnectValidate the key against GET /tasks, not /health (health ignores the key).
MethodPathNote
GET/tasksfilter ?completed, page ?limit&offset — no total count
POST/tasks→ 201
PATCH/tasks/{id}full replacement
POST/tasks/{id}/completeone-way — no reopen endpoint
DELETE/tasks/{id}→ 204, needs a confirm dialog

ACT 2

Task list — filters & Load more

↔ 9d01446

The primary screen: URL-driven filters and pagination with no total count.

claude — frontend/
Build /tasks with a .task-card list and .badge
priority/status (icon + label, never color alone).
Filter All/Active/Completed in the URL (?status=).
Pagination is Load more: append, raise offset,
hide when a short page returns, reset on filter
change. Complete in place; Delete via a .dialog
confirm. loading / empty / error states, aria-live.
  • Filter state lives in ?status=active
  • Complete → POST /tasks/{id}/complete
  • Delete → accessible <dialog> → 204
⚠️
GotchaNo total → Load more, hidden on a short page, reset on filter change. No numbered pages.

✔ Verify — filter chips change the URL & list; Complete moves a task; Delete asks first.

ACT 3

Create — shared form & validation

↔ 1eb8c15

A reusable <TaskForm> and validation that matches the backend exactly.

claude — frontend/
Build /tasks/new → POST /tasks (201). Extract a
reusable <TaskForm> + <FormField> on the .form-field
classes, shared with edit. Title required (trim;
block whitespace-only before any request). Priority
default medium. Disable submit in flight.
On 422, render the detail STRING inline —
never detail[0].msg.
  • Trim & block whitespace-only titles client-side
  • No double POST — disable while submitting
  • Reuse the same form for edit
⚠️
Gotcha422 detail is a string. detail[0].msg throws.

✔ Verify — a blank title is blocked before any call; a forced 422 renders its string inline.

ACT 4

Task detail — only real actions

↔ 2aaf787

Model the screen as explicit states, and expose only capabilities the API has.

claude — frontend/
Build /tasks/:taskId → GET /tasks/{id} as a
discriminated union: loading | loaded | notfound |
error. Show id, title, description, badges. Actions:
Edit, Complete (only when active), Delete
(confirm), Back. No "Reopen" action.
Unknown id → 404 → "Task not found" + link back.
  • State machine, not scattered booleans
  • Complete hides once done
  • 404 is a first-class state
⚠️
GotchaNo Reopen — completion is one-way; the backend has no un-complete endpoint.

✔ Verify — a completed task shows no Reopen; /tasks/999999 → "Task not found."

ACT 5

Edit — the full-replacement PATCH

↔ 7a30a7d

The subtlest rule of the day, plus a small shared-hook refactor.

claude — frontend/
Build /tasks/:taskId/edit → PATCH /tasks/{id},
reusing <TaskForm> in edit mode (load, pre-fill).
PATCH is a FULL replacement: submit the
entire TaskInput every time — never partial —
and never send or touch completed.
Extract a shared useFlash hook; reuse on list+detail.
  • Load first, then pre-fill the fields
  • Whole object on every save
  • useFlash shared across screens
⚠️
GotchaSend only the changed field and PATCH silently wipes the rest. Never include completed.

✔ Verify — edit only the title → description & priority are preserved, not reset.

ACT 6

Mid-session 401 — reset the session

↔ d313a63

Handle a key that stops working after you connected — revoked or expired.

claude — frontend/
Add setUnauthorizedHandler to the client. On any
/tasks* 401 with a stored key: clear the key
(memory + sessionStorage), record a reason, and
redirect to /connect, which shows that reason.
Remember a 401 has no request_id — don't
rely on it here.
  • Global handler in the client, not per-screen
  • Clear + redirect, don't just show an error
  • Reason surfaced on the Connect page
⚠️
GotchaShow an in-place error and the user is stuck on a screen that can't load. Reset the session.

✔ Verify — change the backend key mid-session → the next action bounces to /connect with the reason.
